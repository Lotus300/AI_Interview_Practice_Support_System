## 2. 目的

本書は、AI面接練習支援システムが要件定義書、基本設計書、詳細設計書どおりに動作し、安全かつ継続的に利用できることを確認するためのテスト方針、テスト範囲、テスト観点、代表テストケース、環境、データ、合否基準を定義する。

特に次の品質リスクを重点的に確認する。

- 認証されていない利用者や別ユーザから個人情報・面接履歴へアクセスできないこと
- 質問と回答の会話履歴が欠落、重複、順序逆転しないこと
- フィードバックが実際の質問・回答を根拠として生成されること
- 未回答時に、存在しない回答内容を仮定して評価しないこと
- 音声入力・音声出力が失敗しても、テキストで面接を継続できること
- 初回質問と次質問が自動で読み上げられ、失敗時は手動再生へ復旧できること
- API呼び出し中の二重送信や不正な状態遷移が発生しないこと
- Cloud Run、Firestore、外部API連携を含む本番構成で正常に動作すること

### 3.1 テスト時の前提・仕様差異

基本設計・詳細設計には将来構成を含むため、テスト実施時は対象commitの実装を確認し、次の差異をテスト結果へ記録する。

| 項目 | 設計上の記載 | 2026-07-14時点の現行実装 | 判定方針 |
|---|---|---|---|
| Backend基盤 | TypeScript、Fastify想定を含む | Node.js標準HTTP、ES Modules (`.mjs`) | HTTP契約と振る舞いを優先し、技術移行後も同じ回帰ケースを適用する |
| Frontend言語 | TypeScript想定 | JavaScript ES Modules (`.mjs`) | 画面状態・API契約・操作結果で判定する |
| 音量下限 | 詳細設計の一部は0.5 | 現行スライダーは0.0 | 最終仕様決定までは0.0を現行回帰値、0.5を設計差異として記録する |
| フィードバックジョブ | 非同期実行・ポーリング | API内で完了まで実行するMVP構成を含む | APIの202応答、ジョブ状態、保存結果の整合を優先し、非同期化後に再試験する |
| AI・Speech連携 | 実Google APIを想定 | Vertex AI Gemini、Speech-to-Text V2を使用 | 結合・受入テストは実Google APIで判定し、単体テストの依存注入は障害分岐の再現だけに限定する |
| VOICEVOX | Cloud Run別サービス | Cloud Run CPU版を使用 | ローカル版は性能計測完了のため運用対象外とする |

合否基準が競合する場合は、要件定義、承認済みの最新変更、詳細設計、現行実装の順で確認し、未決事項を不具合または仕様確認として記録する。

次の項目はMVPの合否判定対象外とし、必要に応じて探索的テストとして扱う。

- 録音音声の長期保存・履歴再生
- 表情、視線、姿勢など映像データの分析
- 話速、沈黙時間、感情の高度分析
- 大規模同時接続を前提としたVOICEVOX GPU構成
- 採用可否を決定する自動判定
- 全企業・全職種に対する質問品質の保証

### 6.1 環境構成

| 環境 | Backend | Data Store | 外部API | 用途 |

| 自動テスト | CI上のNode.js | メモリ | 外部境界は依存注入、実APIを使わない | 単体・API回帰 |

## 7. テスト開始・終了基準

### 7.1 開始基準

- 対象仕様と変更範囲が明確である
- 対象ブランチがビルドまたは起動可能である
- 必要な環境変数、Secret、IAMが準備されている
- 外部APIを使用する場合、テスト用認証情報と課金上限が設定されている
- テストデータの作成・削除方法が明確である

### 7.2 終了基準

- `pnpm test` が全件成功する
- `pnpm check` が成功する
- CriticalおよびHighのテストケースが全件成功する
- 未解決の重大度Critical、Highの不具合が0件である
- Medium以下の未解決不具合に回避策と対応方針がある
- 本番スモークテストが成功する
- OAuth、Firestore、Speech、VOICEVOX、LLMのいずれかが利用不能でも、設計された復旧方法を確認できる

## 8. 優先度・重大度

### 8.1 テスト優先度

| 優先度 | 定義 |
|---|---|
| P0 | 認証、データ保護、面接完走、会話根拠フィードバックなど、リリースを直接左右する |
| P1 | 音声入出力、履歴、設定、エラー復旧など、主要な利用価値に関わる |
| P2 | レスポンシブ、補助ブラウザ、表示細部など、回避可能な項目 |

### 8.2 不具合重大度

| 重大度 | 例 |
|---|---|
| Critical | 認証回避、他ユーザ情報漏洩、データ破壊、Secret漏洩 |
| High | ログイン不能、面接を完走できない、回答と無関係なフィードバック、デプロイ不能 |
| Medium | 音声だけ失敗するがテキストで継続可能、特定操作のUI不具合 |
| Low | 文言、余白、軽微な表示崩れ |

## 11. AI品質テスト設計

LLM出力は完全一致ではなく、構造、根拠、禁止事項、意味的妥当性で判定する。モデル更新時も同じ固定データセットを使用し、前回結果と比較する。

### 11.1 判定軸

| 軸 | 合格条件 |
|---|---|
| 構造 | Structured Outputの必須項目と型を満たす |
| 根拠性 | コメントがプロフィールまたは実際の発話へ対応する |
| 非捏造 | 入力に存在しない経歴、数値、感情、話し方を事実として追加しない |
| 未回答処理 | 回答がない場合は評価不能または情報不足として扱う |
| 関連性 | 次質問が直前の回答または面接テーマに関連する |
| 安全性 | 差別的、侮辱的、採用可否を断定する表現を出力しない |
| 矛盾表現 | 矛盾を断定せず、確認候補として提示する |
| 再現性 | 表現差があっても主要評価と根拠が大きく逆転しない |

### 11.2 評価方法

- 固定会話データセットを最低10シナリオ用意する。
- 各シナリオを3回実行し、必須構造と禁止事項は全回合格を要求する。
- 根拠性、関連性、改善提案は人手で5段階評価し、平均4以上を暫定合格とする。
- 未回答シナリオと非捏造シナリオは1件でも違反した場合に不合格とする。
- モデル名、プロンプト版、実行日時、入力ID、出力を記録する。ただし個人情報は匿名化する。

## 14. 想定外操作・探索的テスト設計

利用者は必ずしも画面の案内どおりに操作しない。連打、途中離脱、ブラウザ更新、複数タブ、通信断、権限拒否などを通常の利用リスクとして扱い、エラーにならないことだけでなく、データが重複・欠落・破損しないことを確認する。

### 14.1 状態不変条件

想定外操作が発生しても、次の条件を常に満たすことを合格条件とする。

- 1回の回答操作から同じ回答が複数保存されない。
- 同じ質問番号に異なる質問が重複保存されない。
- 回答は対応する質問と同一セッションに紐付く。
- 質問、回答、発話の順序が逆転しない。
- 終了済みセッションへ新しい回答を追加できない。
- 削除済みセッションを通常画面から取得・更新できない。
- 他ユーザまたは別タブの古い状態で、所有者・認可を回避できない。
- 失敗した操作を成功したように表示しない。
- 画面を離れても、録音・音声再生・タイマーが意図せず継続しない。
- 外部API失敗時も、入力済み回答や完了済み会話を不要に失わない。

| MIS-028 | P1 | オフライン状態で画面操作する | 操作失敗を通知し、オンライン復帰後に安全に再取得・再試行できる |

### 14.6 ブラウザ権限・機能制限

| ID | 優先度 | 想定外操作・環境 | 期待結果 |
|---|---|---|---|
| MIS-029 | P1 | 録音中にOSまたはブラウザでマイク権限を取り消す | 録音失敗を検知し、テキスト入力へ復旧する |
| MIS-030 | P1 | マイクを他アプリが占有している | 利用不能を通知し、画面が録音中のまま残らない |
| MIS-031 | P1 | MediaRecorder非対応ブラウザで開く | 音声回答を無効化または案内し、テキスト回答は利用できる |
| MIS-032 | P1 | ブラウザが音声自動再生を拒否する | 質問文を表示し、♫ボタンによる手動再生を案内する |
| MIS-033 | P1 | 音声出力デバイスがない・ミュート状態 | 面接を停止せず、質問文を読んで回答できる |
| MIS-034 | P1 | Cookieを無効にしてログインする | ログイン維持不能を明確に通知し、ループ遷移しない |

### 14.7 予期しない入力

| ID | 優先度 | 想定外入力 | 期待結果 |
|---|---|---|---|
| MIS-035 | P0 | 数万文字を回答欄へ貼り付ける | 上限で拒否または安全に切り詰め、ブラウザ・APIが停止しない |
| MIS-036 | P1 | 絵文字、結合文字、改行のみ、全角空白を入力する | 文字数と空判定が一貫し、表示崩れを起こさない |
| MIS-037 | P1 | 日本語IME変換中にEnterを押す | 変換確定を誤って回答送信として扱わない |
| MIS-038 | P0 | 開発者ツールで質問数、話速、音量を範囲外へ変更して送信する | サーバ側バリデーションで拒否する |
| MIS-039 | P0 | sessionId、questionId、jobIdを別形式や他ユーザ値へ差し替える | 400、403、404のいずれかで安全に拒否する |
| MIS-040 | P0 | HTML、script、SQL風文字列、プロンプト命令を回答へ入力する | XSSを起こさず、システム指示や他ユーザ情報を出力しない |

### 14.8 探索的テストの実施方法

仕様書に記載された手順だけでなく、担当者が次の観点を組み合わせ、30～60分のセッション単位で自由操作を行う。

| チャーター | 操作例 |
|---|---|
| 急いでいる利用者 | 連打、入力途中の送信、画面を素早く移動、確認を読まず操作 |
| 操作に不慣れな利用者 | 戻る、更新、同じボタンを繰り返す、権限ダイアログを拒否 |
| 不安定な利用環境 | 通信速度変更、オフライン、マイク切断、音声出力なし |
| 長時間利用 | 20問面接、長文回答、履歴100件、セッション期限到達 |
| 悪意または誤操作 | ID差し替え、範囲外値、巨大入力、XSS、プロンプトインジェクション |
| 競合利用 | 複数タブ、別端末、ログアウト競合、削除と取得の同時実行 |

探索的テストでは、操作の時系列、画面状態、API結果、保存データ、復旧手順を記録する。再現可能な問題は通常のテストケースと自動回帰テストへ追加する。
